## 1. 产品定位（北极星）

**一个虚拟艺人的专业级 AI 创作工作室。**

玩家先用对话式访谈「捏」出一个虚拟艺人，让 ta 看起来像个真实存在的人；然后用**专业制片级的工作流**为这个艺人持续产出五类作品：陪伴对话、写真/视频、音乐、访谈成片、以艺人为演员的短片/短剧。

每一条生产线都对标该领域**业内最专业的软件 / SaaS / AI 应用**——不是玩具级一键黑盒，而是有形象一致性、分阶段可控、资产管理、质量档位的专业工作流。

**与旧定位的根本区别**：彻底去掉经营模拟（资金、行动点、热搜、危机、世界引擎、日结/周结算）。本产品不是经营游戏，是**创作工具**。艺人是中枢，作品是产出，没有公司经营层。

**继承 M1**：能力网关、6 个云端 Provider 适配器、异步 job 队列、成本闸门、用量账本、健康面板、设置页（key 后台录入）全部复用，不重写。

## 2. 让产品「专业」的五条横切标准

这是把本产品和玩具区分开的技术脊梁，**所有引擎共享**：

### PS-1 形象一致性（核心壁垒）
每个艺人维护一个**一致性参考包**：定妆照集（参考图）+ 声线档案（描述 + 后续 TTS 音色映射）+ 视觉/性格文字描述。所有写真、视频、短剧、配音生成**强制引用**该参考包，跨所有作品锁定同一张脸、同一把声音、同一种气质。这是专业级 AI 创作与随机生成的根本差异，也是 v3 规格 AI-7 的落地。

### PS-2 分阶段可控工作流
每条生产线是**专业制片流程**而非整体黑盒：每个中间阶段（音乐蓝图 / 剧本 / 分镜 / 配音…）都可预览、可单点重抽（re-roll 单格分镜而不重做整集）、可人工编辑。失败在阶段级别暴露原因，不是整体翻车。

### PS-3 资产与版本管理
所有产出带元数据入库（艺人 / 类型 / 创建时间 / 提示词 / 版本 / 来源阶段），可浏览、收藏、对比、跨线复用（音乐工坊产出可当短剧主题曲，定妆照可当视频首帧）。资产持久化为 JSON 索引 + 落盘媒体文件。

### PS-4 质量档位与成本透明
每条重媒体生产线提供两档：**草稿/试听档**（快、省，用于探索）与**发行档**（精，用于成品）。生成前展示预估成本并需确认（复用 M1 的 confirm 闸门 CL-6），用量入账本（GW-7）。

### PS-5 异步任务与进度可视
所有重媒体生成走 M1 统一 job 队列（GW-6）：提交即返回 jobId、阶段级进度、持久化、重启恢复、失败带原因可重试。前端统一「生成中」卡片，专业级进度反馈。

## 3. 五条引擎的专业级需求

> 对标即「按这些专业工具的核心工作流与质量标准打造引擎架构」。最终成片质量取决于所接入的 Provider 与本机 ffmpeg（见 §7 诚实边界）。

### E1 对话陪伴（对标 Character.AI / Replika）
- **永不出戏**：全艺人档案（人设/背景/性格/说话风格）注入系统提示词，回复始终在角色内。
- **长期记忆**：跨会话记住与玩家的关系、重要往事、近期作品；记忆持久化，不止于单会话上下文。
- **情绪与关系演化**：艺人有随互动变化的心情/亲密度状态，影响回复语气。
- **流式输出**：打字感（M4 的流式，提前到此引擎）。
- **（可选）主动消息**：艺人可在特定条件下主动发起对话。

### E2 写真 / 视频（对标 Midjourney / Runway / Kling 专业流）
- **参考图锁脸**：定妆照作身份锚，所有生成引用一致性参考包。
- **可控生成**：构图 / 姿态 / 场景 / 景别 / 比例（竖屏 9:16 等）控制；提示词自动增强（注入视觉档案）。
- **批量变体与单图重抽**：一次出多张候选，可对单张重抽。
- **视频**：定妆照为首帧的图生视频，运镜描述，时长档。
- **画廊式资产库**：按艺人归档，元数据可检索，收藏。

### E3 音乐（对标 Suno 专业档）
- **两段式生产**：先出可确认的作曲蓝图（歌名/曲风/BPM/调式/分段歌词/制作建议），确认后才渲染。
- **完整曲式**：主歌/副歌/桥段结构；歌词编辑器可改。
- **声线匹配**：渲染参数匹配艺人 voiceProfile（性别声线）。
- **试听档 vs 发行档**：低成本快速试听 / 高质量发行（替代 v2 的 45 秒 clip/正式版本地档位）。
- **作品库**：按艺人归档，可带入短剧当主题曲。

### E4 访谈成片（对标专业视频制片管线）
- **分阶段管线**：策划（嘉宾画像+分段问题脚本）→ 录制/ASR 转写 → 包装（亮点/金句）→ TTS 配音（音色匹配艺人）→ 字幕烧录（SRT）→ ffmpeg 时间线拼接 → MP4。
- **双人画面**：主持人 + 嘉宾参考图合成。
- **（可选）口型对齐**：依赖对应 Provider。
- **进度可视**：script/audio/visual/subtitle/final 五阶段逐步更新。

### E5 短片 / 短剧（对标竖屏短剧工业）
- **完整管线**：立项（题材/集数/单集时长/主演=艺人）→ 分集剧本（场景/动作/对白/情绪/钩子，结构化 JSON 可编辑）→ 选角定妆（配角 AI 生成人设+定妆照，主演用艺人一致性包）→ 分镜（每场景分镜图，可单格重抽）→ 图生视频片段（竖屏 9:16）→ 分角色配音+字幕 → 成片合集（单集 MP4 + 连播）。
- **一致性锁定**：主演跨全剧锁脸锁声（一致性参考包）。
- **两档模式**：低成本（静帧+运镜 ken-burns+配音，视频用量减 80%）/ 高质量（全 AI 视频片段）。
- **成本闸门**：每集生成前展示预估，整剧批量逐集或一次确认。

## 4. 里程碑分解（艺人为中心，逐个交付）

每个子里程碑走完整的「设计→计划→子代理实现+双评审→合并」闭环，做完合并再开下一个。依赖顺序从下往上。

| 里程碑 | 内容 | 引擎 | M1 能力 | 依赖 |
|---|---|---|---|---|
| **S1 艺人创设 + 档案 + 一致性包** ⭐ | 对话式访谈捣人 → 完整艺人档案 → 定妆照集（一致性参考包 PS-1） | — | `content`/`image` | M1 |
| **S2 艺人对话室（陪伴 E1）** | Character.AI 风格 in-character 对话、长期记忆、情绪演化、流式 | E1 | `chat` | S1 |
| **S3 写真 + 视频生产线（E2）** | 锁脸写真、定妆照首帧视频、可控生成、画廊资产库 | E2 | `image`/`video` | S1 |
| **S4 音乐生产线（E3）** | 两段式蓝图→渲染、声线匹配、作品库 | E3 | `content`/`music` | S1 |
| **S5 访谈成片（E4）** | 策划→ASR→包装→TTS→字幕→ffmpeg 拼接 | E4 | `content`/`asr`/`tts`/`video` | S1/S3 |
| **S6 短片/短剧（E5）** | 立项→剧本→选角→分镜→视频→配音字幕→成片连播 | E5 | 全部 | S1/S3/S4 |

里程碑顺序可在每个交付后按反馈调整；S5/S6 是重媒体管线，引入 ffmpeg（CL-3 允许的纯 CPU 工具）。

## 5. S1 详细设计：艺人创设 + 档案 + 一致性参考包

S1 是整个产品的地基——所有引擎都挂在「艺人」上。S1 交付：艺人数据模型、对话式访谈捣人流程、定妆照生成（一致性参考包雏形）、艺人管理 UI。

### 5.1 艺人档案数据模型

持久化为 `data/artists.json`（艺人数组，JSON 文件存档，延续 M1 零依赖风格）。每个艺人：

- **身份**：`id`、`name`（中/英艺名）、`gender`、`persona`（人设，如"冷艳"）、`positioning`（定位，如"电子歌手"）
- **真实感内核**（"看起来像活人"的关键，AI 自动生成）：`backstory`（成长经历/出身/转折）、`personality`（性格特质数组，喂给对话室保持不出戏）、`coreAppeal`（核心吸引力）、`speakingStyle`（说话风格）
- **一致性参考包**（PS-1，跨生产线复用）：
  - `voiceProfile`：声线文字描述 + `ttsVoice`（预留 TTS 音色映射，S1 可空）
  - `visualIdentity`：外貌文字描述（注入所有图像/视频提示词）
  - `musicStyle`：音乐风格倾向（喂给 S4）
  - `portraits`：定妆照对象数组 `[{url, prompt, createdAt}]`
- `createdAt`、`updatedAt`、`schemaVersion`

档案 schema 在服务端集中定义并校验（缺字段/类型错 → 结构化错误）。

### 5.2 对话式访谈捣人流程

1. **访谈循环**（沉浸感）：玩家点「创设新艺人」，AI 以访谈系统提示词驱动，一次问一个问题（想要怎样的艺人？性格？音乐风格？外貌气质？艺名想法？），玩家边答边聊。前端累积对话历史。走 `chat` 或 `content` 能力。
2. **生成档案**（结构化抽取）：玩家点「根据对话生成档案」，后端把访谈记录喂给 `content` 能力（严格 JSON schema），产出**完整结构化艺人档案草稿**——AI 自动补全背景故事/性格/声线/视觉描述等字段。
3. **确认 / 微调**：草稿展示在可编辑表单，玩家改任意字段，点「确认创建」保存进 `data/artists.json`。
4. **生成定妆照**：创建后（或玩家点「出定妆照」）调 `image` 能力，用 `visualIdentity` 作提示词生成 1~2 张定妆照，追加进 `portraits`（图片落 `prototype/generated/`，已 gitignore）。

> 设计要点：访谈（自由对话）与结构化抽取（生成档案）**分两个端点**，职责清晰——访谈管沉浸感，生成管 schema。无 key 时两者返回 `unconfigured`（不崩，沿用 M1 约定）。

### 5.3 API（沿用 M1 路由分发与错误约定）

- `POST /api/artist/interview` — body `{messages:[]}` → `{reply, provider, model}`（访谈下一问）
- `POST /api/artist/finalize` — body `{transcript}` → `{draft}`（结构化档案草稿）
- `POST /api/artist` — body `{profile}` → 校验后保存，返回 `{id, artist}`
- `GET /api/artists` → `{artists:[...]}`（列表，含定妆照缩略）
- `GET /api/artist/:id` → `{artist}`
- `PUT /api/artist/:id` — body `{profile}` → 更新（微调已存档案）
- `DELETE /api/artist/:id` → `{ok}`
- `POST /api/artist/:id/portrait` — body `{stylePrompt?}` → 生成定妆照，追加到 portraits

### 5.4 前端

工作台新增**「艺人」主区**（成为首屏）：
- 创设对话界面：聊天气泡（访谈）+「生成档案」按钮 + 档案编辑卡（可改字段）+「确认创建」+ 定妆照展示
- 已创设艺人卡片列表：头像（定妆照）/ 艺名 / 人设，点入详情，可删除、可「出定妆照」
- M1 的六能力测试卡降级为可折叠「调试」区；健康 / 设置 / 成本面板保留（配 key 用）

所有动态内容沿用 M1 的 `esc()` HTML 转义；media 用 `mediaHtml`。

### 5.5 数据与持久化

- `data/artists.json`：艺人档案数组（`data/` 已 gitignore）。读写经服务端，原子写（先写临时文件再 rename，避免半写损坏）。
- 定妆照媒体落 `prototype/generated/`（M1 的 `ctx.saveFile`），档案只存 url。
- schema 带 `schemaVersion`，为后续里程碑加字段留迁移空间。

### 5.6 测试策略（沿用 M1）

- **单测**（`node --test`）：档案 schema 校验（缺字段/类型/空名拒绝）、artists.json 原子读写、finalize 的 JSON 解析容错（AI 返回非纯 JSON 时的提取）、portrait url 追加、路径安全。
- **冒烟**（扩展 `scripts/smoke.mjs`）：无 key 时 interview/finalize → `unconfigured`；artist 创建→列表→详情→删除 CRUD 走通；坏输入（空 profile/空 name）→ 400/bad_request。

## 6. 架构与非功能（继承 M1）

- **零 npm 依赖**、Node ≥18 ESM、原生 `http`、静态 `prototype/` 前端、端口 3100。
- 艺人相关逻辑放 `src/studio/`（与 `src/gateway/` 同级），档案读写 `src/studio/artists.js`，访谈/抽取提示词构造 `src/studio/artist-create.js`。新端点追加到 `src/api/routes.js`。
- **安全**（NFR-4）：key 仅服务端、路径防穿越、`no-store`、`data/` 与 `.env` 不入库。
- **可观测**（NFR-3）：AI 调用失败入 `logs/`（M1 账本/错误日志已有）。
- 应用层错误一律 HTTP 200 + `{error:{code,message}}`；400 仅 JSON 解析失败，404 未知路由。

## 7. 诚实边界

- 引擎的**架构与工作流**按专业标准打造（一致性、分阶段、可控、可重抽、资产管理）；**最终成片质量取决于所接入的 Provider 与本机 ffmpeg**。口型对齐、发行档视频、整曲音乐等需对应付费 API + ffmpeg。
- 当前环境 **keyless**（6 个 Provider 全 unconfigured）：所有引擎逻辑可开发、可跑通降级路径，但真实出片需在设置页配 key。
- Suno 音乐适配器形态 **UNVERIFIED**（按 sunoapi.org 社区文档实现），接入真实 key 时需校准。
- S5/S6 的 ffmpeg 依赖为本机 CPU 工具（CL-3），需用户本机安装；不依赖 GPU。
